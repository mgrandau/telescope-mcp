# Journal - December 13, 2025

## Origin Story

Retroactive journal entry. This project predates the Human-AI Intent Transfer Principles. It began as a port from an older `telescope-control` project and evolved into something fundamentally different — an AI-controlled telescope.

## The Problem

The original `telescope-control` was a collection of Jupyter notebooks for manual telescope operation — camera captures, motor commands, position readings, all run cell-by-cell. It worked, but every observing session required a human stepping through notebooks in the right order, adjusting settings by hand, and manually correlating what the camera showed with where the telescope was pointing.

The idea: what if an AI agent could operate the telescope? Not just "take a picture" but the full workflow — discover cameras, capture frames, slew to targets, monitor conditions, manage observing sessions. The agent needs structured tools, not notebooks.

## Key Design Decisions

### MCP over REST API

The telescope is controlled via MCP (Model Context Protocol), not a REST API. This means AI agents (Claude, GPT) can call telescope operations as native tool calls in conversation, not through HTTP requests.

**Rejected alternative:** REST API only. Rejected because REST requires the AI to construct HTTP requests, parse responses, and manage state — all of which MCP handles natively. The AI just calls `capture_frame(camera_id=0)` and gets structured results.

**Why both MCP and HTTP:** The web dashboard runs on HTTP/FastAPI because humans need a browser interface for monitoring and manual control. The key architectural insight: **two control paths (MCP for AI, HTTP for human) share the same device layer.** Neither path owns the hardware — they both go through the same `CameraRegistry`, `SessionManager`, etc.

### Dual control paths sharing a device layer

The MCP server and web dashboard are equal peers, not client-server. Both call into the same device abstractions. This means an AI agent can capture a frame while a human watches the live stream, and they see the same camera state.

**Rejected alternative:** MCP wrapping the web API (AI calls HTTP endpoints). Rejected because it adds latency, couples the MCP tools to the HTTP interface, and means changes to the dashboard break the AI tools.

**Rejected alternative:** separate device instances per control path. Rejected because hardware is shared — you can't have two Camera objects talking to the same physical camera. The registry pattern ensures single ownership with shared access.

### Driver injection (hardware vs digital twin)

Every hardware component has two drivers: a real one (ASI cameras, Arduino sensors, serial motors) and a digital twin (simulation). The `--mode` flag switches between them at startup. Tests always use the digital twin.

**Rejected alternative:** mock objects in tests only. Rejected because the digital twin serves double duty: it's the test fixture AND the development environment. You can develop the entire MCP server, web dashboard, and tool chain without any hardware connected. The digital twin isn't a test utility — it's a first-class runtime mode.

**Why not hardware abstraction layer (HAL)?** The Protocol-based DI pattern IS the HAL, just without the enterprise naming. `CameraDriver` Protocol → `ASIDriver` or `TwinDriver`. Same result, less ceremony.

### AI-optimized over-documentation

Every function — including private methods and test helpers — has full Google-style docstrings (Args, Returns, Raises, Example). This is deliberate: AI agents see docstrings inline when reading source files. Rich docstrings mean the agent doesn't need to traverse multiple files to understand a function's contract.

**Rejected alternative:** minimal documentation (code is self-documenting). Rejected because AI agents work in limited context windows. A 500-token docstring that explains what a function does, what it expects, and what can go wrong is cheaper than the agent reading 50 lines of implementation to figure it out.

**Open question:** As models improve, is this level of documentation still needed? (See docscope-mcp #17 for the same question at the tooling level.)

### ASDF for session storage

Observing sessions are stored in ASDF (Advanced Scientific Data Format), not JSON/CSV/SQLite. ASDF natively handles numpy arrays (camera frames), metadata, and hierarchical data — all things telescope sessions produce.

**Rejected alternative:** JSON files. Rejected because camera frames are numpy arrays. JSON can't store binary data efficiently — you'd need base64 encoding, which bloats file sizes 33% and loses type information.

**Rejected alternative:** SQLite. Rejected because session data is hierarchical (session → exposures → frames + metadata), not tabular. SQLite can store blobs but the schema would be awkward for nested scientific data.

### Architecture READMEs in every subpackage

Every subpackage (`data/`, `devices/`, `drivers/`, `tools/`, `web/`, `observability/`, `utils/`) has its own README with: overview, public surface, dependencies, invariants, and AI accessibility map. Plus a root architecture README with the full system diagram.

**Why so many READMEs?** This is the largest project in the portfolio (80+ files, 2895 statements). A single README can't cover the architecture. The per-package READMEs let an AI agent read only the relevant section — `devices/README.md` when working on camera code, `tools/README.md` when adding MCP tools.

## What This Project Taught Us

1. **Dual control paths are powerful.** Human monitoring + AI operation on shared hardware is a better workflow than either alone.
2. **Digital twins aren't just for testing.** A simulation mode that runs the full stack without hardware makes development possible anywhere — not just in front of the telescope.
3. **AI-optimized docs pay for themselves.** The heavy documentation investment means Copilot and Claude can work on this codebase with minimal guidance.
4. **Hardware projects need more architecture docs, not fewer.** The physical constraints (USB bandwidth, serial protocols, sensor timing) aren't visible in the code — they need to be documented or the next developer (human or AI) will break them.
